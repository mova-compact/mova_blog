# ТЗ на персональный Jekyll-сайт как publishing hub

## 1. Финальное намерение

Создать персональный статический сайт на Jekyll с публикацией через GitHub Pages как основную публичную площадку автора.

Сайт должен одновременно выполнять две функции:
- быть author-friendly publishing infrastructure для систематизации и публикации материалов;
- быть публичной витриной проектов, методик и экспертного присутствия автора.

Результат данного ТЗ считается достаточным, если кодирующий агент может взять его в работу без дополнительных смысловых уточнений, а все вопросы, выходящие за рамки явно определённого, эскалируются пользователю.

## 2. Цель сайта

Сайт создаётся не как визитка и не как узкий блог, а как персональная publishing system / knowledge hub.

Ключевые цели:
- публикация материалов из Git-репозитория в Markdown;
- поддержка нескольких типов контента;
- удобное ручное сопровождение без backend, БД и server-side логики;
- расширяемая архитектура под добавление новых разделов;
- профессиональный, чистый, технологичный образ автора.

## 3. Технологические рамки

Обязательные ограничения:
- SSG: только Jekyll;
- публикация: GitHub Pages как основная площадка;
- контент: Markdown и файловая структура в Git-репозитории;
- без backend, БД и server-side логики;
- структура должна быть Git-friendly и удобной для ручного сопровождения;
- начальная версия должна работать при минимальном количестве контента.

ТЗ должно опираться на стандартные механизмы Jekyll: pages, posts, collections, front matter, front matter defaults, layouts, includes, Liquid templates, `_config.yml`.

## 4. Стратегия архитектуры

Стратегия проектирования: гибрид контентных сущностей и Jekyll-архитектуры.

Порядок принятия решений:
1. Зафиксировать site strategy.
2. Зафиксировать контентные сущности.
3. Построить информационную архитектуру.
4. Привязать сущности к Jekyll-механике.
5. Определить шаблонную логику.
6. Определить workflow публикации и deployment.

Запрещено проектировать сайт только от временного sitemap или только от темы оформления.

## 5. Целевая модель сайта

Сайт должен поддерживать следующие классы материалов:
- длинные статьи;
- короткие заметки или будущие короткие форматы;
- проектные страницы;
- главы личной истории;
- статические страницы;
- потенциальные будущие тематические коллекции.

Архитектура должна допускать рост без полной перестройки: добавление новых разделов должно достигаться через новые collections, layouts, includes, defaults и навигационные конфигурации, а не через переписывание всей модели сайта.

## 6. Контентное ядро старта

На старте гарантированно существуют следующие направления:
- блог / публикации по проекту MOVA;
- личная история, публикуемая главами не в хронологическом порядке;
- раздел проектов.

Начальная публикационная база может быть минимальной:
- экосистема MOVA как первый смысловой кластер;
- первая глава личной истории;
- базовый каркас раздела проектов.

Архитектура главной страницы и навигации должна адекватно работать при очень малом количестве контента.

## 7. Информационная архитектура

### 7.1 Базовые разделы первого релиза

Обязательные разделы первого релиза:
- Home;
- Articles / Blog (или эквивалентный раздел публикаций);
- Story;
- Projects;
- About.

Допустимые будущие разделы без изменения общей модели:
- Research;
- Notes;
- Documentation;
- Resources / Library;
- Topic hubs;
- отдельные серии или коллекции.

### 7.2 Навигационные принципы

Навигация должна:
- давать быстрый доступ к основным публичным областям;
- оставаться короткой и устойчивой при малом стартовом объёме;
- поддерживать расширение без разрушения логики;
- разделять evergreen-страницы, публикации и проектные сущности.

### 7.3 Логика главной страницы

Главная страница не должна быть просто списком последних постов.

Home должна выполнять 4 функции:
- объяснять, кто автор и чем занимается;
- показывать ключевые тематические векторы;
- вести в основные контентные сущности;
- давать вход в проекты и MOVA-экосистему.

## 8. Контентная модель

### 8.1 Сущности

Минимальный набор сущностей:
- `post` — обычная публикация / статья;
- `story_chapter` — глава личной истории;
- `project` — проектная карточка / проектная страница;
- `page` — статическая страница;
- `topic` или аналогичная будущая сущность — для тематических хабов при расширении.

### 8.2 Рекомендуемая реализация в Jekyll

- `post` — стандартные `_posts`;
- `story_chapter` — отдельная collection, например `_story`;
- `project` — отдельная collection, например `_projects`;
- `page` — обычные страницы верхнего уровня или в выделенной структуре;
- будущие сущности — отдельные collections по мере появления устойчивой потребности.

### 8.3 Поля front matter

#### Для `post`
Обязательные поля:
- `title`
- `date`
- `summary`
- `tags`
- `categories`
- `topic`
- `status` (`draft` / `published`)
- `featured` (bool)
- `seo_title` (optional)
- `seo_description` (optional)

#### Для `story_chapter`
Обязательные поля:
- `title`
- `chapter_id`
- `sequence` или `arc_order`
- `summary`
- `status`
- `featured`

Принципиально важно: порядок отображения глав личной истории не должен зависеть только от даты публикации.

#### Для `project`
Обязательные поля:
- `title`
- `summary`
- `status`
- `start_date` (optional)
- `project_type`
- `topics`
- `featured`
- `links` (optional)

### 8.4 Front matter defaults

Для снижения ручной нагрузки должны быть предусмотрены front matter defaults по типам сущностей.

Цель defaults:
- уменьшить повторение полей;
- унифицировать layouts и permalink logic;
- сделать добавление нового материала предсказуемым.

## 9. Шаблонная логика Jekyll

### 9.1 Layouts

Минимальный набор layouts:
- `default`
- `home`
- `post`
- `story_chapter`
- `project`
- `page`
- `archive` / `listing`

### 9.2 Includes

Минимальный набор includes / partials:
- `site-header`
- `site-footer`
- `meta`
- `content-card`
- `project-card`
- `story-card`
- `tag-list`
- `breadcrumbs` (optional)
- `related-content`
- `pagination-nav` (если используется)

### 9.3 Listing pages

Должны быть предусмотрены страницы-индексы для:
- публикаций;
- проектов;
- истории;
- архивов / тематических подборок по мере роста.

### 9.4 Перелинковка

Сайт должен поддерживать перелинковку между:
- публикациями и проектами;
- публикациями и MOVA;
- главами истории;
- тематически связанными материалами.

Связность должна строиться через теги, категории, topic-поля и ручные связи в front matter при необходимости.

## 10. URL и permalink-логика

Требования:
- URL должны быть человекочитаемыми и устойчивыми;
- permalink-структура должна различать типы сущностей;
- story chapters не должны смешиваться с обычными постами;
- projects должны иметь собственное пространство URL;
- структура должна быть совместима с будущим ростом и SEO.

Рекомендуемый принцип:
- статьи: `/articles/:slug/`
- история: `/story/:slug/`
- проекты: `/projects/:slug/`
- статические страницы: `/about/`, `/topics/`, и т.д.

## 11. Главная страница

Главная страница должна включать:
- сильный intro-блок о том, кто автор и над чем работает;
- блок навигации по основным направлениям;
- выделение MOVA как стартового центра экосистемы;
- вход в личную историю;
- блок проектов;
- блок последних или избранных публикаций;
- ясный переход в архивные / индексные разделы.

Главная не должна зависеть от большого числа постов и не должна выглядеть пусто при малом стартовом архиве.

## 12. Визуальные требования

Сайт должен выглядеть:
- профессионально;
- чисто;
- технологично;
- небанально;
- не как шаблонный обычный блог.

Визуальный язык должен поддерживать образ автора как человека, работающего с AI, агентными системами, workflow-архитектурой, цифровыми продуктами и собственными методиками.

Предпочтение: restrained editorial-tech aesthetic без ощущения generic template.

## 13. SEO-основа

Обязательно предусмотреть:
- уникальные title/description для ключевых страниц;
- корректные canonical-подходы;
- Open Graph / social metadata;
- sitemap;
- RSS/feed;
- чистые URL;
- базовую схему архивов и тегов/категорий без SEO-мусора;
- человекочитаемые заголовки и summary поля.

SEO должно быть совместимо с GitHub Pages и Jekyll-экосистемой без зависимости от server-side решений.

## 14. GitHub Pages и deployment

### 14.1 Базовый принцип

Публикация должна быть рассчитана на GitHub Pages как основной путь доставки сайта.

### 14.2 Требования к deployment

- сборка должна быть предсказуемой;
- структура проекта должна быть пригодна для размещения в GitHub-репозитории;
- deployment workflow должен быть документирован;
- допустимо использовать GitHub Actions, если это упрощает и стабилизирует сборку;
- решение должно оставаться понятным для ручного сопровождения.

### 14.3 Требования к репозиторию

Репозиторий должен быть организован так, чтобы:
- было понятно, где лежит контент;
- было понятно, где лежат layouts/includes/assets/data;
- добавление нового материала не требовало знания внутренней магии темы;
- структура была прозрачной для кодирующего агента и пользователя.

## 15. Рекомендуемая структура репозитория

```text
/
├─ _config.yml
├─ _data/
├─ _includes/
├─ _layouts/
├─ _posts/
├─ _projects/
├─ _story/
├─ assets/
│  ├─ css/
│  ├─ js/
│  ├─ img/
│  └─ fonts/ (optional)
├─ pages/
│  ├─ about.md
│  ├─ projects.md
│  ├─ story.md
│  └─ articles.md
├─ index.md
├─ Gemfile
└─ README.md
```

Допустимо уточнение структуры агентом, если он не меняет сущностную модель и соблюдает рамки ТЗ.

## 16. Редактирование и публикация новых материалов

Процесс публикации должен быть простым и предсказуемым.

Требования:
- новый материал создаётся как файл в соответствующей сущности;
- front matter должен быть стандартизован;
- layouts и defaults должны минимизировать ручную настройку;
- статус публикации должен быть понятен;
- draft/published logic должна быть прозрачной;
- публикация через Git должна быть понятна без сложной операционной рутины.

## 17. Требования к росту

Архитектура должна заранее учитывать возможность последующего добавления:
- новых разделов;
- мультиязычности;
- тегов и категорий;
- RSS;
- архивов;
- тематических хабов;
- дополнительных коллекций контента;
- richer internal linking.

При этом стартовая реализация не должна быть перегружена функциями, не нужными на первом этапе.

## 18. Границы полномочий кодирующего агента

Кодирующий агент обязан:
- реализовывать сайт строго в рамках данного ТЗ;
- не менять смысловую модель сайта самостоятельно;
- не вводить дополнительные сущности без основания в ТЗ;
- обращаться к пользователю за подтверждением при любых существенных неоднозначностях.

Кодирующий агент может самостоятельно:
- выбрать конкретную внутреннюю файловую организацию;
- определить частные naming conventions;
- предложить удобную реализацию layouts/includes/workflow;
- уточнить build/deploy mechanics, если они не противоречат ТЗ.

## 19. Неопределённости, признанные допустимыми

На момент старта допустимо не фиксировать окончательно:
- полный перечень будущих разделов;
- финальную рубрикацию;
- точную языковую стратегию;
- полный набор будущих типов контента;
- долгосрочный ритм публикации.

Эти зоны должны трактоваться как открытые параметры роста, а не как разрешение исполнителю самостоятельно изменить ядро архитектуры.

## 20. Критерии приёмки ТЗ

ТЗ считается пригодным к передаче в работу, если по нему однозначно определимы:
- цель сайта;
- роль сайта;
- обязательные разделы первого релиза;
- стартовые контентные сущности;
- соответствие сущностей механизмам Jekyll;
- правила расширяемости;
- логика главной страницы;
- шаблонная структура;
- базовая SEO-логика;
- deployment-подход;
- границы решений, которые агент может принимать сам;
- список вопросов, которые требуют подтверждения пользователя.
